# Deployment Guide

How to deploy the Agent Orchestrator to production.

| Domain | Service |
|--------|---------|
| `https://agents-orchestrator.com` | Dashboard (FastAPI + WebSocket) |
| `https://monitoring.agents-orchestrator.com` | Grafana (metrics dashboards) |

## Architecture

```
Browser
  ├── agents-orchestrator.com:443
  │     → Nginx (SSL, rate limiting, security headers)
  │       → Dashboard :5005 (FastAPI + WebSocket)
  │         → PostgreSQL :5432
  │         → Redis :6379
  │
  └── monitoring.agents-orchestrator.com:443
        → Nginx (SSL, security headers)
          → Grafana :3000
            → Prometheus :9090
```

## Prerequisites

| Requirement | Details |
|-------------|---------|
| AWS EC2 | Amazon Linux 2023 or Ubuntu 22.04, **t3.medium spot** (2 vCPU, 4 GB RAM, ~$9.50/mo) |
| Elastic IP | Static public IP associated to the EC2 instance |
| DNS A records | `agents-orchestrator.com` + `monitoring.agents-orchestrator.com` → Elastic IP |
| Security Group | TCP 22 (SSH, your IP), TCP 80 (HTTP), TCP 443 (HTTPS) |
| GitHub OAuth App | Created at github.com/settings/developers |
| OpenRouter API key | From openrouter.ai |

## Step 1: AWS Setup

### EC2 Instance

1. Launch an EC2 instance (t3.small minimum, Amazon Linux 2023)
2. Allocate an **Elastic IP** and associate it to the instance
3. Configure **Security Group**:
   - Inbound: TCP 22 (your IP), TCP 80 (0.0.0.0/0), TCP 443 (0.0.0.0/0)
   - Outbound: all traffic

### DNS

Create two A records:

```
agents-orchestrator.com            → A → <Elastic IP>
monitoring.agents-orchestrator.com → A → <Elastic IP>
```

Verify with `dig agents-orchestrator.com` and `dig monitoring.agents-orchestrator.com`.

### GitHub OAuth App

1. Go to **github.com → Settings → Developer settings → OAuth Apps → New OAuth App**
2. Set:
   - **Homepage URL**: `https://agents-orchestrator.com`
   - **Authorization callback URL**: `https://agents-orchestrator.com/auth/github/callback`
3. Note the **Client ID** and **Client Secret**

## Step 2: EC2 Setup

SSH into the instance and run the setup script:

```bash
scp scripts/setup-ec2.sh ec2-user@<EC2_IP>:~/
ssh ec2-user@<EC2_IP>
chmod +x setup-ec2.sh && ./setup-ec2.sh
```

The script installs Docker, creates the app directory (`/opt/agent-orchestrator`), and generates `.env.prod` with random secrets.

## Step 3: Configure Environment

Edit the generated `.env.prod`:

```bash
nano /opt/agent-orchestrator/.env.prod
```

Fill in the values:

```env
ENVIRONMENT=production
BASE_URL=https://agents-orchestrator.com

# PostgreSQL (auto-generated, keep as-is)
POSTGRES_PASSWORD=<auto-generated>

# Auth
JWT_SECRET_KEY=<auto-generated>
OAUTH_CLIENT_ID=<from GitHub OAuth App>
OAUTH_CLIENT_SECRET=<from GitHub OAuth App>

# API Keys
OPENROUTER_API_KEY=<from openrouter.ai>

# Grafana (auto-generated)
GRAFANA_PASSWORD=<auto-generated>

# GitHub
GITHUB_USERNAME=<your GitHub username>

# Sandbox (optional — per-session isolated execution)
SANDBOX_ENABLED=true
SANDBOX_TYPE=docker
SANDBOX_IMAGE=python:3.12-slim
SANDBOX_MAX_CONCURRENT=10
```

## Step 4: Deploy

### First deployment

```bash
cd /opt/agent-orchestrator
git clone <repo-url> .

# Start services (without SSL first — for certbot ACME challenge)
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d postgres redis dashboard grafana prometheus nginx
```

### Obtain SSL Certificates

```bash
# Request certificates for both domains
docker compose -f docker-compose.prod.yml run --rm certbot certonly \
  --webroot -w /var/www/certbot \
  -d agents-orchestrator.com -d monitoring.agents-orchestrator.com \
  --agree-tos --email your@email.com --non-interactive

# Restart Nginx to load the certificates
docker compose -f docker-compose.prod.yml restart nginx
```

### Start All Services

```bash
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d
```

### Verify

```bash
# Dashboard health check
curl -s https://agents-orchestrator.com/health

# Grafana
curl -s https://monitoring.agents-orchestrator.com/api/health

# All containers
docker compose -f docker-compose.prod.yml ps
```

## Step 5: GitHub Actions (CI/CD)

The deploy pipeline (`.github/workflows/deploy.yml`) auto-deploys on push to `main` (ignores `docs/`, `*.md`, `terraform/`).

### Pipeline Steps

1. **Test Suite** — Install deps (Python 3.12), run `pytest`, lint with `ruff`
2. **Deploy to EC2** — rsync code, inject secrets into `.env.prod`, rebuild containers, health check

Full chain: `test → lint → rsync code → inject secrets → build → deploy → health check`.

### Deploy Strategy

The deploy uses `docker compose down` (full teardown) instead of selective `rm` to prevent stale container name conflicts. An additional cleanup step force-removes any orphaned containers with the `agent-orchestrator` prefix before starting services fresh.

```
rsync code → inject secrets → build images → down (full stop) → rm stale containers → up -d → health check
```

### Secret Injection

All GitHub Secrets are injected into `.env.prod` on EC2 via the `_inject()` helper (idempotent upsert). Secrets managed:

- `AWS_*`, `OPENROUTER_API_KEY`, `JWT_SECRET_KEY`, `OAUTH_CLIENT_ID/SECRET`
- `GRAFANA_SMTP_*`, `POSTGRES_PASSWORD`, `BASE_URL`, `GITHUB_USERNAME`

### Force-Recreate Scope

Only `dashboard` and `aws-cost-exporter` are force-recreated on deploy. Stateful services (postgres, redis, nginx, tempo) are preserved between deploys to keep data/TLS state.

### Operational Notes

- **Tempo**: trace backend container (ports 3200 Tempo API, 4318 OTLP HTTP, 7-day retention). Defined in `docker-compose.prod.yml`.
- **`OTEL_EXPORTER_OTLP_ENDPOINT`**: set on the `dashboard` container (e.g. `http://tempo:4318`) to enable trace export. Omit to disable tracing (graceful no-op).
- **Postgres password sync**: `ALTER USER` runs on every deploy to fix first-init password mismatch.
- **Nginx timeout**: 600 s (10 min) for long team runs.
- **`BASE_URL`**: `https://agents-orchestrator.com` (domain, not IP — required for OAuth callbacks).
- **Static cache busting**: bump `?v=NNN` in `index.html` on every frontend change.

## SSL Certificate Renewal

Certificates auto-renew via the `certbot` container (checks every 12h). To manually renew:

```bash
docker compose -f docker-compose.prod.yml run --rm certbot renew
docker compose -f docker-compose.prod.yml restart nginx
```

## Monitoring

- **Grafana**: `https://monitoring.agents-orchestrator.com` (login: admin / `GRAFANA_PASSWORD`)
- **Prometheus**: `127.0.0.1:9090` (localhost only, access via SSH tunnel)

### Grafana Dashboards

| Dashboard | Description |
|-----------|-------------|
| Orchestrator | Overview: requests, tokens, cost, status |
| API Calls & Costs | Per-model API call volume and cost |
| Errors & Alerts | Error rates, alert history |
| Agent Usage | Agent interactions and delegation |
| Cost Analysis | OpenRouter cost vs agent correlation |
| Infrastructure | EC2 host + container metrics |
| AWS Costs | Cost Explorer: MTD, forecast, per-service |
| AWS S3 Storage | Bucket size, objects, requests, bandwidth, latency |

### S3 Monitoring

The S3 dashboard requires **request metrics** enabled on S3 buckets. The exporter auto-enables these at startup (`PutMetricsConfiguration` with ID `EntireBucket`). Storage metrics (size, object count) are always available. Request metrics (GET/PUT counts, errors, bandwidth, latency) appear ~15 minutes after being enabled.

Required IAM permissions: `s3:PutMetricsConfiguration`, `s3:ListAllMyBuckets`, `cloudwatch:GetMetricStatistics`.

```bash
# Prometheus (raw metrics, SSH tunnel)
ssh -L 9090:localhost:9090 ec2-user@<EC2_IP>
# Open http://localhost:9090
```

## Security

| Layer | Protection |
|-------|-----------|
| **Transport** | TLS 1.2+, HSTS, strong ciphers |
| **Auth** | GitHub OAuth2 + JWT sessions, API keys (dashboard) |
| **Grafana auth** | Admin password, sign-up disabled |
| **Rate limiting** | 10 req/s per IP on dashboard (Nginx) |
| **Headers** | X-Frame-Options, X-Content-Type-Options, Referrer-Policy |
| **Secrets** | No secrets in code, `.env.prod` excluded from git |
| **Metrics** | `/metrics` blocked externally (403) |
| **CI/CD** | pip-audit, CodeQL, Trivy, TruffleHog scans |

## Sandbox (Code Execution)

Per-session isolated Docker containers for agent-generated code. Each session gets its own container with port forwarding for live preview.

**Prerequisites**:
- Docker socket must be accessible from the dashboard container (`/var/run/docker.sock` is mounted in `docker-compose.prod.yml`)
- The dashboard Dockerfile includes Docker CLI installation

**Environment variables** (in `.env.prod`):

| Variable | Default | Description |
|----------|---------|-------------|
| `SANDBOX_ENABLED` | `false` | Enable sandbox system |
| `SANDBOX_TYPE` | `docker` | `docker` (production) or `local` (testing) |
| `SANDBOX_IMAGE` | `python:3.12-slim` | Default Docker image for sandbox containers |
| `SANDBOX_MAX_CONCURRENT` | `10` | Max simultaneous sandbox containers |

**Port forwarding**: Sandbox containers expose services on host ports 9000-9099. In production, Nginx proxies these via `/sandbox-preview/{port}/` so no extra EC2 security group rules are needed.

**Dashboard APIs**:
- `GET /api/sandbox/status` — system overview
- `GET /api/sandbox/{id}/info` — container status, ports, uptime
- `GET /api/sandbox/{id}/logs` — SSE log streaming
- `DELETE /api/sandbox/{id}` — force stop container
- `WS /ws/sandbox/{id}/terminal` — interactive shell

**Security note**: The Docker socket grants root-equivalent access to the host. In production, ensure the EC2 instance is properly isolated and only trusted users have dashboard access. See `docs/security.md` for details.

## Container Runtime: OrbStack

Docker containers (Postgres, dashboard, docs) run on **OrbStack** — not Docker Desktop. Same `docker` / `docker compose` CLI. Tests and linting run locally via Python venv.

- Container startup: **0.2 s** (vs 3.2 s Docker Desktop) — **16× faster**
- Idle RAM: ~180 MB (vs 2+ GB) — **11× less memory**

All environments (local dev, CI, production) use OrbStack. Never fall back to Docker Desktop.

## Troubleshooting

### Dashboard not loading

```bash
docker compose -f docker-compose.prod.yml logs dashboard --tail 50
docker compose -f docker-compose.prod.yml logs nginx --tail 50
```

### Grafana not loading

```bash
docker compose -f docker-compose.prod.yml logs grafana --tail 50
```

### SSL certificate issues

```bash
docker compose -f docker-compose.prod.yml run --rm certbot certificates
docker compose -f docker-compose.prod.yml run --rm certbot renew --force-renewal
docker compose -f docker-compose.prod.yml restart nginx
```

### Database connection

```bash
docker compose -f docker-compose.prod.yml exec postgres pg_isready -U orchestrator
docker compose -f docker-compose.prod.yml exec postgres psql -U orchestrator
```

### WebSocket not connecting

Check that the Security Group allows TCP 443 and that Nginx WebSocket config has long timeout (`proxy_read_timeout 86400`).

### Deploy fails with container name conflict

If deploy fails with `Error when allocating new name: Conflict. The container name ... is already in use`:

```bash
# SSH into EC2 and force-remove stale containers
ssh ec2-user@<EC2_IP>
docker ps -a --filter 'name=agent-orchestrator' --format '{{.ID}}' | xargs -r docker rm -f
cd /opt/agent-orchestrator
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d
```

This happens when Docker leaves containers with prefixed names (e.g. `21d9af4e7301_agent-orchestrator-*`) that `docker compose rm` doesn't clean up. The CI pipeline now handles this automatically with a forced cleanup step.
